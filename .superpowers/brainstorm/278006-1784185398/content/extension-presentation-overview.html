WIDI Extension Presentation Protocol

第一版只支持预定义、transport-neutral 的展示语义。Extension 说明“是什么”，consumer 决定“怎么画”。全部 review 裁决已并入各通道定义与边界规则：Status 快照查询与生命周期、notify/diagnostic 硬边界、persistent 的 entryId 去重、human request 隐私留痕、presentationId 作为 output/notification 的稳定 view key。

Extension Core

emitOutput(text)
追加执行过程文本；ephemeral
setStatus(key, status) / clearStatus(key)
keyed 当前状态；status = { text, progress? }
notify(text)
瞬时提醒；仅 info 语义，无 code、无 attention
publishMessage(message)
持久化展示消息；text / markdown / code；返回 session entryId
reportDiagnostic(draft)
结构化问题事实；severity / disposition / code / message / details
requestHuman(draft)
唯一需要 consumer 返回响应的通道
→

WIDI Core Ownership

自动绑定 agentId / extensionId；extension 不可伪造 attribution
自动生成 presentationId：output / status / persistent message / notification 的 runtime event key；persistent hydration 去重另用 entryId
commandId 关联：line-command execution 将 commandId 传给 runner 的 command context；observer/interceptor/tool context 不携带 commandId，事件中该字段可选缺席
status registry：(agent, extension, key) → status + updatedAt，提供 listExtensionStatuses(agentId) 快照查询
persistent message：先写 session（core:extension_message custom entry），拿到 entryId 后再发布 event；永不进入 model context
diagnostic：复用 CoreDiagnostic，domain=extension，source 注入，local code 规范化为 extension.<extensionId>.<code>
顺序契约：顺序 await 的 presentation 调用位于 command_accepted 与 command_completed 之间，按调用顺序到达
validation、size limits、reload/dispose cleanup（含 status registry 清空）
主动发布的 presentation 不回灌 extension observers
→

Consumer Projection

TUI：output/persistent/diagnostic → timeline；notify → transient notice；status → working-status 槽位 + neutral chips（仅 active agent）；human request → capturing overlay + 隐私安全留痕
CLI：统一的有界纯文本降级；每通道映射为确定性单行格式（见下）
RPC：转发同一 canonical event；status 由 query 补齐晚接入快照
Headless：记录或忽略展示事件
第一版无 capability negotiation，无 ctx.ui，无 Component 穿越边界
未来可选 TUI presentation entry 只做增强，不参与 core 行为

六类预定义通道（含全部裁决）

1. Output

emitOutput(text: string): Promise<void>

命令或任务执行中的追加式文本。完成痕迹由 extension 自己 emit（如 “✓ Index build completed”），不由 status 折叠产生。

presentationId 是 output item 的稳定 view key，commandId 关联所属命令，均由 core 生成注入。

ephemeral append-only timeline 已实现（含 presentationId / commandId）

2. Status

setStatus(key: string, status: { text: string; progress?: { completed: number; total?: number }; }): Promise<void> clearStatus(key: string): Promise<void>

可替换、可清除、可查询的当前状态。progress 只有最小形状——不加 percent、stage 数组，等真实需求再扩；total 缺席表示总量未知，consumer 画 spinner 而非进度条。绝不进入 timeline，也不携带 severity：chip 一律 neutral，warning/error 色彩与 attention 只来自 diagnostic。

registry 顺序契约与 agent_status_changed 相同：set 先写 registry 再发 event，clear 先删除再发 event，listener 内查询必须看到新值；晚接入 / hydration 中的 consumer 用 listExtensionStatuses(agentId) 重建 status area，不回放事件。extension reload/dispose、agent dispose 时 core 清空条目并发 clear event；command 结束不自动清除——清除是 extension 的显式责任，consumer 可显示 status 年龄（updatedAt）。

core registry replace/clear snapshot query 已实现（含 lifecycle / validation / CLI）

3. Notification

notify(text: string): Promise<void>

一次性提醒，仅 info 语义：无 severity、无 code、无 dedupe、不产生 attention。任何“有问题”的事实必须走 reportDiagnostic——这条硬边界防止六通道退化为“作者随便挑一个能显示的”。显示时长与清除由 consumer 决定。

ephemeral fire-once notice 已实现（4 KiB / commandId / CLI）

4. Persistent Message

publishMessage(message: { kind: "text" | "markdown" | "code"; title?: string; content: string; }): Promise<{ entryId: string }>

可恢复的展示内容。以 core:extension_message custom entry 落 session，hydration 时重建；永不进入 model context。Action 返回值、live event 与 hydration 都携带同一个 entryId：projector 用它消除快照与 buffered event 竞争产生的重复。

session append-only timeline core 已实现（含 entryId / validation / CLI）

5. Diagnostic

reportDiagnostic(draft: { severity: "info" | "warning" | "error"; disposition?: "reported" | "degraded"; code: string; message: string; details?: Record<string, unknown>; }): Promise<void>

复用 CoreDiagnostic：domain=extension，source/agentId/extensionId 由 core 注入；local code 被规范化为 extension.<extensionId>.<code>。disposition 缺省为 reported，作者不能自称 blocked。该 action 产生的 diagnostic 不回灌 extension observers。

每次上报生成新的 core id，是独立事实：dedupe 只合并同一事实的多路径观察（throw / return / event），不合并重复上报。持续性问题由 attention 的持续性表达，作者不应轮询式重报同一 code。

diagnostic flow fresh id attention 已实现（fresh id / validation / attribution）

6. Human Request

requestHuman(draft: HumanRequestDraft): Promise<HumanResponse>

唯一需要 consumer 返回响应的交互通道。HumanRequestEnvelope 已携带 broker 解析的 agentId，不依赖 pending event 补身份。TUI 以 capturing overlay 呈现，按键走 tui.select.* keybinding map。

resolved 后追加 ephemeral 折叠留痕：confirm 显示 Yes/No；select 的答案 ∈ options 时显示该项（候选文本本就展示过），free input 回退 “Answered”；input / argumentsCompletion / custom 一律只显示 “Answered”，不回显潜在敏感值。

request broker response overlay 已实现（含 envelope.agentId）

Canonical event 与身份契约

extension_output { presentationId, agentId, extensionId, commandId?, text, createdAt } extension_status_changed { presentationId, agentId, extensionId, commandId?, key, status?: { text, progress?: { completed, total? } }, changedAt } status 缺席表示 clear；真实当前值以 core registry 为准。 extension_notification { presentationId, agentId, extensionId, commandId?, text, createdAt } extension_message_published { presentationId, entryId, agentId, extensionId, commandId?, message: { kind, title?, content }, createdAt } entryId 是 live/hydration 稳定身份；presentationId 只标识本次 runtime event。 diagnostic { diagnostic: { id, domain: "extension", code, severity, disposition, agentId, extensionId, source, message, details? }, createdAt } human_request_pending { agentId, request: HumanRequestEnvelope // envelope.agentId 与外层一致 } human_request_resolved { agentId, requestId, response, completedAt } human_request_timeout / human_request_cancelled { agentId, requestId, reason?, completedAt }

同一个 agent 中的 TUI 形态

status/progress 只出现在 working-status 槽位与 neutral chips，不进 timeline；“✓ 完成”行来自 extension 自己的 emitOutput；diagnostic 与 notify 分开标注；human request 是 capturing overlay，resolved 后只留一行 ephemeral、隐私安全的折叠记录。

不可突破的边界

无 UI 反向依赖
Core extension 不 import pi-tui，不读取当前 consumer 类型。
无隐式 renderer
command result 的对象字段不会被解释成按钮、widget 或布局。
展示事件只读
交互只通过 requestHuman 或已注册 command 完成。
无 presentation EventBus
Extension 主动发布的展示事实不回灌 observers。
Durability 明确
只有 persistent message 写 session；status 仅是 runtime current state；两者都永不进入 model context。
Payload bounds ≠ flood control
core 对单条 text/message/details 施加 size limit；第一版不做事件速率限制。pi-tui requestRender() 只合并约 16ms 内的绘制请求，不减少 event 数、projector 工作量或 timeline 内存；CLI 与 component formatter 继续做有界截断。
降级确定性
没有自定义 presentation module 时，所有 consumer 仍能正确表达语义；CLI 的每通道单行格式是契约，不是实现细节。
Attribution 不可伪造
agentId / extensionId / presentationId / entryId 全部由 core 生成注入。
