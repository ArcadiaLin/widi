WIDI TUI 交互设计稿 v2：全状态与命令落实

基于 v1 实现（apps/widi-pi/src/tui）的现状盘点 + 本版新增裁决。标记约定： 现状 已实现；提案 本版新增设计；待裁决 需要产品决定；v1 裁决 被本版修订的原裁决。

悬浮层只留给必须立即响应的交互
pi-coding-agent 用悬浮 selector 做命令/会话选择；WIDI 不跟随。凡是“从候选中选一个”的应用内选择，一律使用 editor 上方的 inline completion-menu：它在正常 component flow 中推高 transcript，不遮挡历史，可以边选边对照上下文。
已裁决：human request 一律由 completion-menu 承接，不再使用 overlay
v1 裁决（§12）让所有 human request 走 capturing overlay；/login 的 onSelect 暴露了遮挡问题。本版修订：confirm/select/input 统一由 editor 上方的 completion-menu 呈现（配色与样式沿用现状），overlay 只保留给导致程序无法运行的致命错误。core broker 契约不变。见「Human Request」节。
两个菜单的位置互不侵占
Editor 打字触发的 autocomplete 在 editor 下方展开（pi-tui 现状）；承接 needs-argument 与 human request 的 completion-menu 持续展开在 editor 上方，两者可同时存在、互不替换。
展示类事实进 timeline / notice，不抢焦点
auth_login_url / code / progress、extension notify 都是展示类事实：timeline application-notice 或 transient notice，绝不打开任何选择界面。
命令是唯一交互词汇
所有能力（含 skill、prompt template）都经 CommandEngine 的 line/inline 命令触达；任何命令缺少参数时，一律用 completion-menu 展开可用选项（needs-argument 是全命令统一行为，不是个别命令的特例）。

6d. overlay 的新职责：只承接致命错误 已裁决

┌─ ✕ WIDI cannot continue ─────────────┐ │ orchestrator.spawn_failed │ │ Extension "audit" failed to │ │ activate: missing config key… │ │ ▸ Quit View diagnostics │ └───────────────────────────────────────┘ fatal overlay：模态、必须表态；不再承接任何 human request。

已裁决 overlay 只保留给「导致程序无法继续运行」的报错事件，筛选后仅两类：(1) 启动失败——runtime 或初始 agent 创建失败、不存在任何可用 agent，程序无法完成任何实际工作；(2) 未捕获的致命 application error——application 边界捕获的 unexpected failure，状态可能损坏，不应继续。两者都以模态 overlay 呈现错误摘要与 code，操作为 Quit / View diagnostics（后者关闭 overlay 只读查看诊断视图）。明确排除：后台 agent spawn 失败（strip 呈现）、hydration 失败（notice）、disposeAll 失败（正在退出）、单个 command 失败（command result 行）。human request 不再有 overlay fallback——menu host 在 spawn 前就绪，任何请求都有承接处。

后台请求不全局弹出：升该 agent 的 human-request attention；若 menu 已开则 tab 行出现新项，否则在 status 槽位上方提示待答请求数。已裁决 增加 app.request.open 快捷键（可配置），直接跳答最近的后台请求。

关键推论：menu 必须是多步交互宿主，不是一次性选择框
登录是「多次交互串联」的典型：select 登录方式 → 展示授权链接 → 等待用户输入 code（或 callback 自动成交）。因此 completion-menu 需要支持原地状态变换：同一条 login 流程的连续请求复用同一槽位，内容随流程推进 replace-in-place（select → 展示 URL/设备码 + 输入行 → waiting/provisional），而不是每步开关一次菜单。waiting 态的 menu 显示 spinner 与说明文字并保持输入行可用；provisional 请求被 core 撤下时 menu 静默关闭、焦点归还 editor。设备码流程同理：menu 内嵌展示 userCode + verificationUri（来自 auth_login_code 的事实），输入行等待确认。多步内容仅存在于 menu 生命周期内，resolved 后只有隐私安全的一行进 transcript（§6c）。
